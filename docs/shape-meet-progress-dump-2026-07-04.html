Alineacion de producto. La app debe entenderse como una app de videollamadas profesional con una capa de identidad local, no como una herramienta de face swap. El host autorizado entra desde la desktop, configura camara, rostro, voz y fondo, crea una reunion y comparte un enlace. El invitado entra por web desde movil o PC. Que ya existe

Producto y flujo de usuario

  • Desktop Tauri/Rust con UI React para host autorizado y acceso publico para invitados.
  • Pantalla inicial minimalista orientada a reuniones, no a face swap.
  • Login host separado. Ser host es un rango administrado desde panel.
  • Flujo de crear/unirse a reunion, lobby, configuracion de equipo, configuracion de host y sala.
  • Host puede seleccionar identidad, activar rostro, voz y fondo, capturar calibracion de fondo y entrar a llamada.
  • Invitado publico puede abrir enlace web y entrar desde movil o PC.

Admin/API

  • Next.js admin con Prisma/Postgres, sesiones, usuarios, roles, reuniones, participantes, identidades, artifacts y audit log.
  • APIs versionadas por rutas Next: /api/auth/host/*, /api/users/*, /api/admin/identities/*, /api/host/identities/*, /api/meetings/*, /api/livekit/token y /api/health.
  • Flujo de artifactos de identidad con storage persistente, resolucion de URL firmada y validacion SHA256 en smoke remoto.
  • Seed/bootstrap de host/admin para primer deploy, luego RUN_SEED=false.

WebRTC e infraestructura

  • LiveKit elegido como SFU desde temprano para no rehacer signaling al pasar de 1:1 a 3 o 4 participantes.
  • Compose Coolify con admin, Postgres, Redis, LiveKit y coturn.
  • TURN TCP/UDP validado. TURN TLS 5349 omitido por flag en este entorno porque el check actual se concentra en conectividad de demo.
  • Scripts de deploy, env generation, handoff y remote check ya existen y estan documentados en docs/coolify.md.

IA local y sidecar

  • Sidecar Python separado del proceso UI, supervisado por Tauri/Rust.
  • Contrato HTTP para health, diagnostics, pipelines, sessions, frames de video, chunks de audio y preflight.
  • Procesador local empaquetable que puede delegar por comando o por endpoints persistentes.
  • Wrappers CLI para FaceFusion, BackgroundMattingV2 y vcclient000.
  • Model endpoint server preparado para procesos persistentes que mantengan modelos cargados en VRAM.
  • Preflight de contrato pasa con procesadores de desarrollo, pero eso no significa modelos reales activos.

Empaquetado y handoff

  • Workflow y scripts para desktop packages Windows/macOS.
  • Handoff Windows con PowerShell: Build-ShapeMeetWindows.ps1, Test-ShapeMeetWindows.ps1, Install-ShapeMeetAiRuntime.ps1 y Start-ShapeMeetModelEndpoint.ps1.
  • Runtime config desktop con URLs remotas, AI local y Sentry.
  • Debug bundles para recolectar logs utiles en maquinas Windows/NVIDIA.

Decisiones importantes

Decision Estado Razon
LiveKit como SFU Tomada Evita rehacer signaling/rooms/TURN cuando pasemos de 1:1 a maximo 4 participantes.
Desktop Tauri/Rust Tomada Shell nativo para Windows primero, macOS despues, con UI web y sidecar local aislado.
IA local Tomada Host procesa rostro, voz y fondo en su maquina, objetivo RTX 4070+ y premium RTX 4090/5090.
Face swap ruta 1 Condicionada FaceFusion/InSwapper-like es el camino mas rapido para foto fuente, pero requiere resolver licencia comercial si se usa InsightFace/InSwapper en uso comercial.
Ruta 2 preparada Preparada Identidades entrenadas tipo DeepFaceLive/DFM quedan como motor externo futuro, cuidando riesgos GPL.
Ruta 3 documentada Documentada GHOST/GHOST 2.0 queda para I+D posterior por historia legal mas limpia, pero requiere trabajo realtime.
BackgroundMattingV2 Tomada MIT, calidad alta y viable para 720p30 en NVIDIA potente con clean plate.
No camara virtual inicialmente Tomada El producto es una app propia de videollamadas. La ruta de camara virtual puede documentarse despues, no es MVP actual.

Roadmap rapido

  1. Corregir semantica del proyecto: producto final. Mantener fuera de UI publica palabras como "demo", estados internos y textos explicativos. Lo que aun sea diagnostico debe vivir en menu de debug o bundle de soporte.
  2. Preparar runtime NVIDIA reproducible. Crear un instalador/script Windows que instale Python/venv, CUDA-compatible deps, FaceFusion, BackgroundMattingV2, checkpoint BMV2 y vcclient000, luego genere shape-ai-runtime.env.
  3. Conectar FaceFusion real por foto. Hacer que una identidad subida desde admin llegue al host, se cachee localmente y active el wrapper real. Validar 720p30 o degradacion controlada.
  4. Conectar BackgroundMattingV2 real. Implementar y validar flujo de clean plate, reemplazo de fondo, calidad de borde y fallback si no hay calibracion.
  5. Conectar vcclient000 real. Definir endpoint estable de chunks, activar voice toggle, medir latencia y mantener sync con la publicacion LiveKit.
  6. Publicar tracks procesados en LiveKit. Confirmar que el host no solo previsualiza procesado, sino que publica audio y video procesado a los invitados web.
  7. Empaquetar Windows para pruebas. Generar instalador Tauri Windows con runtime config remoto, sidecar Windows y scripts de diagnostico. Probar en PC AMD y luego RTX.
  8. End-to-end real. Host en NVIDIA crea reunion, aplica rostro, voz y fondo, comparte link; invitado abre desde web movil/PC y ve/oye la identidad procesada.
  9. Fortalecer operacion. Sentry releases/sourcemaps, debug bundle automatico, metricas de FPS/latencia/VRAM visibles solo en diagnostico y guias de soporte.

Proximo goal recomendado

El siguiente goal deberia ser un vertical slice real: "Host Windows/NVIDIA publica una reunion Shape Meet con rostro, voz y fondo procesados localmente, y un invitado web lo recibe."

Para avanzar rapido, no conviene abrir mas superficie visual salvo bugs evidentes. El foco debe ser runtime real, publicacion de tracks procesados y empaquetado Windows con diagnostico. La UI debe seguir viendose terminada, pero la prioridad tecnica ahora es reemplazar passthrough por motores reales.
